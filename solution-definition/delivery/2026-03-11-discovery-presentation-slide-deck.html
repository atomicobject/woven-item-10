01

The Problem

Requirements arrive scattered, change without warning,
and are consolidated entirely manually.

02
Context

Two Requirements Problems at Woven

Top-Down

Requirements flowing from Toyota via the CA liaison team into Woven are scattered across tools, poorly structured, and change without any notification reaching the right teams.

Externally constrained, harder to solve. Confirmed priority by Julia Pralle.

Bottom-Up (Internal)

Autonomous driving software was written before formal requirements were established. Knowledge is tribal and scattered inside Woven.

Lower priority — internal to Woven, more available solutions. Out of scope for this effort.

03
Pain Point

Where Requirements Live Today

No single source of truth. No centralized visibility.

📁

Git Repositories

SWRDs as markdown/YAML. Primary source for Lane Change. Not directly surfaced — accessed via PDM docs.

📋

JAMA

Requirements management platform. Ground truth — but populated manually by copy-paste from Git.

📄

Google Docs

PRD and URD documents from CA/PDM team. Nick reviews these daily alongside Git files.

🌐

Confluence

Additional documentation. No systematic connection to JAMA or change tracking.

📊

Spreadsheets

URD spreadsheets from CA. Japanese + English descriptions — used for context only.

💬

Tribal Knowledge

TBDs only resolvable by developers — who often don't know the answer either.

04
Why This Matters

The Scale of the Problem

There are hundreds of requirements across many different functional areas — and every single one is being handled manually today.

Lane Change alone has 600+ SWRDs. Each must be found, read, interpreted, and entered into JAMA by hand — then tracked for changes, linked to test cases, and updated whenever Toyota revises them. There is no automated alert when something changes mid-quarter. There is no single place to see what's been processed and what hasn't.

The problem isn't that the work is too hard — it's that the work doesn't have to be entirely manual.

11
Constraints

What We Can't Change

CA Authorship Is Out of Scope

Hannah and Nick can highlight issues with incoming requirements, but cannot fix them at the source. The quality of what CA writes is outside Woven's control.

We're Not Changing the PRD Process

The tool slots into the existing PRD-driven intake flow. It does not redesign how requirements are scoped, prioritized, or handed off — that process stays as-is.

Teams Won't Move Their Docs

Established practices exist for where requirements live. The solution must work with existing sources, not ideal organization.

Process Still Being Defined

Woven is implementing the CRB process now. The target workflow is not yet stable — but that's an opportunity to shape it.

System 2

Automated Gathering

Instead of hunting through Git and Google Docs one requirement at a time, the tool reads the quarterly PRD and batch-imports every in-scope SWRD at once.

Access tiers — what unlocks at each level
With Git access
Single SWRD Fetch
User pastes a Git link to a specific SWRD. Tool fetches the content directly from the repo and queues it for System 1 — no manual copy from GitHub.
Needs: Git read access
With Git + Google Drive access
User-Initiated Batch
User pastes the PRD link. Tool reads the PRD from Google Drive, follows every SWRD link into Git, and queues them all for System 1 — a full quarter's intake in one action.
Needs: Git + Google Drive read access
With Drive webhook or polling
Auto-Triggered on PRD Approval
When the PRD moves to "approved" in Google Drive, batch import kicks off automatically — no user action needed. Cadence matches the quarterly release cycle.
Needs: Drive webhook or polling + Git access

Key dependency: System 3 is an extension of this — it uses the same access layer to watch for changes. The faster we get Git and Drive access, the faster both Systems 2 and 3 unlock.

System 3

Change Management

Watches Toyota's SWRD files for mid-quarter updates and alerts the right engineers before the impact is missed.

🔍

How It Works

A background job polls the Git repository on a set cadence (nightly, weekly — TBD with Hannah). When a SWRD file changes, it diffs old vs. new and surfaces what specifically changed — interface, specification, or reason section.

Engineers get a digest: which SWRDs changed, what changed, link to the diff. The change can re-trigger System 1 processing for the affected requirement.

⚙️

Access Dependencies

System 3 extends System 2's access layer — it uses the same Git integration to watch for changes rather than batch-import them. No new access is required beyond what System 2 establishes.

Polling cadence and notification channel (Slack, email) are open questions to confirm with Hannah.

Scope may narrow depending on what Reqtify already covers →

Appendix

Glossary

SWRD — Software Requirements Document. Functional requirements from Toyota. Must be reinterpreted for Woven's IPA scope.

SWQT — Software Work/Interface Qualification Test. Test case derived from an SWRD, stored in JAMA with preconditions and test steps.

JAMA — Woven's requirements management system. Authoritative record for internal requirements and test cases.

CA — Customer Applications. Toyota's Japan-based liaison team that translates requirements from TMC to Woven.

CRB — Change Requirement Board. Emerging formal process at Woven to handle and track mid-quarter requirement changes.

URD — User Requirements Document. Parent-level requirement from Toyota; translated into SWRDs by the CA and product teams.

PRD — Product Requirements Document. Quarterly scoped doc listing which SWRDs are in scope for development that quarter.

IPA — Internal Product. Woven's internal product boundary/scope; SWRDs from Toyota must be reinterpreted for IPA level.

Wovey — Woven's internal LLM tool. Hannah uses it for test case drafting (~90% effectiveness, manual curation required).

TSS-4 — Toyota Safety System 4. Legacy rules-based production system; its outdated prescriptive requirements still appear in incoming SWRDs despite Woven's ML approach.

Reqtify — Third-party tool that links artifacts across systems (e.g. requirements in JAMA ↔ test cases). Links existing items — does not move or author content between systems. Potentially complementary to System 3.
